<p>(2nd Part of the 7-Part series on what it takes to be a strong ML engineer in 2026)</p><figure><img alt="" src="/images/blog/part-2-training-and-loss-functions-what-youre-actually-optimizing/hero.jpeg" /></figure><p>Your loss function isn’t just a number that needs to go down. It’s a precise mathematical statement about what you care about and your model will give you exactly what you asked for, even when you asked for the wrong thing.</p><h3>Your loss function is a contract with reality</h3><p>Here’s the pattern I keep seeing in ML failure stories:</p><p>Someone builds a model. It trains beautifully. Loss goes down. Metrics look great. They deploy it.</p><p>Then it fails in production in completely predictable ways — if only they’d actually thought about what their loss function was telling the model to do.</p><p>Real examples I’ve studied:</p><h3>Cross-entropy on imbalanced data:</h3><ul><li>You have 99 normal transactions, 1 fraudulent transaction</li><li>You use standard cross-entropy loss</li><li>You just told your model: “Being wrong on fraud costs the same as being wrong on normal transactions”</li><li>Model learns: “Just predict ‘not fraud’ every time and get 99% accuracy”</li></ul><h3>MSE on heavy-tailed distributions:</h3><ul><li>You’re predicting user engagement (most videos get 100 views, some get 10M)</li><li>You use Mean Squared Error</li><li>MSE heavily penalizes being wrong on outliers</li><li>Model learns: “Optimize for rare viral videos, ignore typical content”</li><li>Your recommendations become completely skewed</li></ul><h3>Standard classification on biased data:</h3><ul><li>Training data has historical bias (fewer loans approved for certain demographics)</li><li>You use cross-entropy to predict “good loan candidate”</li><li>Model learns: “Replicate historical bias perfectly”</li><li>You’ve now automated discrimination at scale</li><li>And it’s mathematically optimal according to your loss function</li></ul><h3>What I’m learning to ask:</h3><p>Before picking a loss function:</p><ul><li>What behavior does this loss actually incentivize?</li><li>How does class imbalance interact with this loss?</li><li>What happens when my data has label noise?</li><li>What are the implicit assumptions in this loss?</li><li>How do those assumptions break in production?</li></ul><h3>Training at scale is distributed systems engineering</h3><p>I used to think training was: write model code, call .fit(), wait.</p><p>Then I tried to actually train something non-trivial and realized: oh no, this is way more complicated.</p><p>What I’m learning about real training:</p><h3>Distributed training isn’t optional at scale:</h3><ul><li>Single GPU: limited by memory and speed</li><li>Multiple GPUs: now you need data parallelism or model parallelism</li><li>Data parallelism: split batches across GPUs, synchronize gradients</li><li>Model parallelism: split the model itself across GPUs</li><li>Pipeline parallelism: different GPUs handle different layers</li><li>Each approach has different communication overhead</li></ul><h3>Gradient accumulation:</h3><ul><li>Want batch size 128 but only fit 32 in memory?</li><li>Accumulate gradients over 4 micro-batches</li><li>Same mathematical result, different memory profile</li><li>But now you need to think about when to synchronize</li></ul><h3>Checkpointing that doesn’t corrupt:</h3><ul><li>Save model state periodically (obviously)</li><li>But also: optimizer state, scheduler state, RNG state</li><li>Handle failures mid-checkpoint</li><li>Don’t fill disk with redundant checkpoints</li><li>Make sure you can actually restore from them (test this!)</li></ul><h3>Reproducibility is harder than it sounds:</h3><ul><li>Random seeds aren’t enough</li><li>GPU operations can be non-deterministic</li><li>Distributed training introduces more randomness</li><li>Data loading order matters</li><li>Even with all this, perfect reproducibility might be impossible</li></ul><h3>Fault tolerance:</h3><ul><li>Your 72-hour training run will crash</li><li>Can you resume from the last checkpoint?</li><li>Do you lose hours of work or just minutes?</li><li>Have you tested your recovery procedure?</li></ul><h3>When to stop:</h3><ul><li>Early stopping isn’t admitting defeat</li><li>It’s recognizing diminishing returns</li><li>That last 0.5% improvement cost you $5K in compute</li><li>Was it worth it? (Usually no)</li></ul><h3>Evaluation: Where theory meets reality (and usually loses)</h3><p>Kaggle metrics ≠ Real-world metrics</p><p>On Kaggle:</p><ul><li>Fixed test set</li><li>Same distribution as training</li><li>Single number to optimize</li><li>Winner takes all</li></ul><p>In production:</p><ul><li>Distribution shifts constantly</li><li>Multiple competing objectives</li><li>Tradeoffs between accuracy, latency, cost</li><li>Need to maintain quality over time</li></ul><p>Offline evaluation ≠ Online performance</p><p>Offline metrics can be completely misleading:</p><ul><li>Great AUC offline, terrible precision at your operating threshold</li><li>High accuracy overall, terrible on rare but important cases</li><li>Perfect on your test set, useless on real user queries</li></ul><p>Online metrics are what actually matter:</p><ul><li>Did users click more?</li><li>Did they convert more?</li><li>Did they come back?</li><li>Did the system make money?</li></ul><p>The gap between offline and online:</p><ul><li>Offline: 95% accuracy</li><li>Online: 70% user satisfaction</li><li>Why? Your test set didn’t capture real user behavior</li></ul><p>What I’m learning to do:</p><p>Build regression test suites for models:</p><ul><li>Like unit tests for code, but for model behavior</li><li>“Model should never predict class A when feature X &gt; threshold”</li><li>“Model should maintain &gt;90% accuracy on this critical subgroup”</li><li>“Model latency should stay under 300ms at p99”</li></ul><p>Understand when metrics lie:</p><ul><li>Average accuracy hides subgroup failures</li><li>Macro-averaging hides class imbalance</li><li>Micro-averaging hides minority performance</li><li>Overall metrics hide temporal degradation</li></ul><p>Track what matters to users:</p><ul><li>Not just “did the model predict correctly”</li><li>But “did this prediction help the user accomplish their goal”</li><li>Sometimes a “wrong” prediction is more useful than a “correct” one</li></ul><h3>What production ML engineers told me</h3><p>“Your offline metrics will lie to you. Build systems that detect this quickly.”</p><p>“Most model failures are gradual, not catastrophic. You need monitoring that catches slow degradation.”</p><p>“The loss function is where you encode your actual business objective. Get this wrong and everything else is pointless.”</p><p>“Training once is easy. Training reproducibly, at scale, with fault tolerance — that’s the actual job.”</p><p>“Your test set is the most important dataset you have. If it’s wrong, all your metrics are meaningless.”</p><h3>Tomorrow: Part 3</h3><p>LLMs and Modern ML — the new fundamentals that everyone’s talking about but few people actually understand deeply.</p><p>What I’m learning about tokenization, attention, context limits, and why RAG systems fail quietly.</p><p>What’s your biggest confusion about training ML models?
