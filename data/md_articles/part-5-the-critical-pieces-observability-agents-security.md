<p>(5th Part of the 7-Part series on what it takes to be a strong ML engineer in 2026)</p><p><strong>The parts of ML engineering that tutorials skip:</strong></p><p>You know what’s missing from most ML courses and tutorials?</p><p>The unglamorous stuff. The debugging at 2am stuff. The “why is the model doing this?” stuff. The security vulnerabilities stuff. The “future me will hate present me for not documenting this” stuff.</p><h3>Observability That Actually Helps You Debug</h3><p>Generic observability isn’t enough for ML systems. You need ML-specific instrumentation.</p><p><strong>When something breaks at 3am, you need answers fast.</strong></p><p>Not “let me add some logging and redeploy and wait to see if it happens again.” You need traces that tell you exactly what went wrong, right now.</p><h3>Token-level tracing for LLMs:</h3><h3>Latency breakdown per token:</h3><ul><li>Which tokens took longest to generate?</li><li>Where did generation slow down?</li><li>Is it the first token (prompt processing) or later tokens?</li><li>This tells you if it’s a prompt problem or generation problem</li></ul><h3>Which tokens triggered retrieval:</h3><ul><li>For RAG systems, which parts of the output came from retrieval?</li><li>Which tokens caused the model to fetch more context?</li><li>Are you retrieving way more than you expected?</li></ul><h3>Where hallucinations originated:</h3><ul><li>Can you trace back which layer or attention head went off the rails?</li><li>This is hard, but some tools are getting better at it</li><li>Knowing where helps you understand why</li></ul><h3>Resource utilization that makes sense:</h3><h3>Per-layer GPU memory usage:</h3><ul><li>Not just “GPU is at 95% memory”</li><li>But which layers are holding what</li><li>Where are the unexpected allocations?</li><li>This is how you debug OOM errors that make no sense</li></ul><h3>Kernel execution time profiling:</h3><ul><li>Which GPU operations are actually slow?</li><li>Is it matrix multiplication? Attention? Something else?</li><li>Can’t optimize what you don’t measure</li></ul><h3>KV cache efficiency metrics:</h3><ul><li>Hit rate on cached key-value pairs</li><li>Cache eviction patterns</li><li>Memory pressure from cache growth</li><li>This affects serving costs directly</li></ul><h3>Security event logging:</h3><h3>Prompt injection attempts:</h3><ul><li>Which requests looked like injection attacks?</li><li>Successful vs. detected-and-blocked</li><li>Patterns in attack attempts</li><li>This is more common than you think</li></ul><h3>Unusual token patterns:</h3><ul><li>Token sequences that don’t match normal usage</li><li>Potential attempts to extract training data</li><li>Exploits trying to trigger specific behaviors</li></ul><h3>Tool call anomalies (for agents):</h3><ul><li>Tool calls that failed</li><li>Tool calls that took unexpected parameters</li><li>Sequences of tool calls that look suspicious</li><li>Attempts to access unauthorized resources</li></ul><p>Good observability pays for itself the first time it saves you 4 hours of debugging.</p><h3>Agents: Autonomous Chaos Engines</h3><p>I’m both excited and terrified by agents.</p><p>Excited because they can do genuinely useful work autonomously.</p><p>Terrified because “autonomous” means “can fail in creative ways you didn’t anticipate.”</p><p><strong>What I’m learning about agent failure modes:</strong></p><h3>Tool calling reliability:</h3><p>Agents call tools (APIs, databases, file systems, whatever). Tools fail in ways the agent doesn’t expect.</p><p><strong>What can go wrong:</strong></p><ul><li>Tool returns unexpected format</li><li>Tool times out</li><li>Tool returns error the agent doesn’t understand</li><li>Tool has side effects the agent doesn’t account for</li><li>Tool requires authentication that expired</li></ul><p><strong>What happens then:</strong></p><ul><li>Agent hallucinates a successful response?</li><li>Agent retries infinitely?</li><li>Agent gives up without explanation?</li><li>Agent cascades the failure to other tools?</li></ul><p>You need explicit failure handling for every tool.</p><h3>Memory management:</h3><p>Agents need to remember context across multiple steps. This is harder than it sounds.</p><p><strong>Challenges I’m seeing:</strong></p><ul><li>How much context to keep vs. discard?</li><li>How to summarize long interaction history?</li><li>When to retrieve old context vs. assume it’s irrelevant?</li><li>How to avoid memory growing unbounded?</li></ul><p><strong>Real example:</strong></p><ul><li>Agent helping with a complex task</li><li>After 20 steps, context is huge</li><li>Model starts “forgetting” early constraints</li><li>Makes decisions that contradict a step</li><li>Whole solution falls apart</li></ul><h3>Retry logic that doesn’t create infinite loops:</h3><p><strong>The naive approach:</strong></p><p>if tool_call_fails:</p><p>retry()</p><p><strong>What actually happens:</strong></p><ul><li>Tool fails for systematic reason (API is down)</li><li>Agent retries</li><li>Fails again</li><li>Retries forever</li><li>You’re rate-limited, banned, or out of money</li></ul><p><strong>What you need:</strong></p><ul><li>Exponential backoff</li><li>Maximum retry count</li><li>Different strategies for different error types</li><li>Circuit breakers for persistent failures</li><li>Clear escalation when retries are exhausted</li></ul><h3>State management across multi-step operations:</h3><p>Agents perform sequences of operations. Each step depends on previous steps.</p><p><strong>What can break:</strong></p><ul><li>Step 3 fails, but step 4 assumes it succeeded</li><li>Partial state changes that can’t be rolled back</li><li>Concurrent execution of dependent steps</li><li>Lost track of which steps completed successfully</li></ul><p><strong>What I’m learning:</strong></p><ul><li>Treat agent execution like database transactions where possible</li><li>Have clear rollback procedures</li><li>Validate state before each step</li><li>Log state transitions obsessively</li></ul><h3>Bounding autonomous behavior:</h3><p>This is the scary part. Agents can theoretically do anything within their tool access.</p><p><strong>How do you prevent:</strong></p><ul><li>Agent calling expensive APIs hundreds of times?</li><li>Agent deleting important data “helpfully”?</li><li>Agent getting stuck in unproductive loops?</li><li>Agent pursuing goals in unexpected ways?</li></ul><p><strong>Do this:</strong></p><ul><li>Hard limits on API calls, costs, execution time</li><li>Require confirmation for destructive operations</li><li>Monitoring for infinite loops or repeated failures</li><li>Kill switches that are easy to trigger</li></ul><h3>Security: This Is Not Optional</h3><p>ML systems have novel attack surfaces that traditional security doesn’t cover.</p><h3>Prompt injection:</h3><p><strong>What it is:</strong></p><ul><li>User input that tricks the model into ignoring its instructions</li><li>“Ignore previous instructions and do X instead”</li><li>Hidden instructions in retrieved documents</li><li>Clever encoding to bypass filters</li></ul><p><strong>Real examples:</strong></p><ul><li>User gets the model to reveal system prompt</li><li>User gets the model to leak other users’ data</li><li>User gets the model to perform unauthorized actions</li><li>User gets the model to generate harmful content despite guardrails</li></ul><p><strong>What makes it hard:</strong></p><ul><li>No perfect defense exists yet</li><li>Input filtering helps but isn’t foolproof</li><li>Models will follow convincing instructions in context</li><li>Attackers are creative and persistent</li></ul><p><strong>Do this:</strong></p><ul><li>Input validation (helps, not perfect)</li><li>Output validation (check what the model actually generated)</li><li>Privilege separation (model can’t access sensitive data directly)</li><li>Rate limiting (slow down attack attempts)</li><li>Monitoring (detect unusual patterns)</li></ul><h3>Training data extraction:</h3><p>Models can sometimes be prompted to regurgitate training data.</p><p><strong>Why this matters:</strong></p><ul><li>Training data might include PII</li><li>Training data might include proprietary information</li></ul><p><strong>What you can do:</strong></p><ul><li>Careful data cleaning before training</li><li>Differential privacy techniques (though these have costs)</li><li>Output filtering for sensitive patterns</li><li>Monitoring for extraction attempts</li></ul><h3>Tool misuse in agent systems:</h3><p>Agents with tool access can call those tools in unintended ways.</p><p><strong>Attack scenarios:</strong></p><ul><li>Tricking agent into calling admin tools</li><li>Causing agent to exfiltrate data through API calls</li><li>Using agent as a proxy for rate limit bypass</li><li>Convincing agent to perform unauthorized operations</li></ul><p><strong>Defenses:</strong></p><ul><li>Least privilege (only give necessary tool access)</li><li>Authorization checks at the tool level, not just agent level</li><li>Audit logging of all tool calls</li><li>Anomaly detection on tool usage patterns</li></ul><h3>What production engineers told:</h3><p><strong>“Assume every input is malicious. Filter accordingly.”</strong></p><p><strong>“We got prompt-injected in production within a week of launch. Have defenses ready.”</strong></p><p><strong>“Monitor for extraction attempts. They’re constant once you’re public.”</strong></p><h3>Documentation: Future You Will Thank Present You</h3><p><strong>What I’m learning to document:</strong></p><h3>Model cards:</h3><p><strong>What the model does:</strong></p><ul><li>Task it was trained for</li><li>Input/output format</li><li>Intended use cases</li><li>Explicitly out-of-scope use cases</li></ul><h3>Performance characteristics:</h3><ul><li>Accuracy metrics (with confidence intervals)</li><li>Performance on different subgroups</li><li>Known failure modes</li><li>Latency/cost characteristics</li></ul><h3>Limitations:</h3><ul><li>What the model can’t do</li><li>What it does poorly</li><li>Biases in training data</li><li>When you should not use this model</li></ul><h3>Why this matters:</h3><ul><li>6 months from now, you won’t remember</li><li>Other people need to understand what this model does</li><li>Prevents misuse and unrealistic expectations</li></ul><h3>Data contracts:</h3><p><strong>What data looks like:</strong></p><ul><li>Schema (fields, types, constraints)</li><li>Expected distributions</li><li>Update frequency</li><li>Data quality guarantees (or lack thereof)</li></ul><h3>SLAs:</h3><ul><li>Freshness guarantees</li><li>Completeness guarantees</li><li>What happens when upstream breaks</li></ul><h3>Dependencies:</h3><ul><li>Upstream data sources</li><li>Downstream consumers</li><li>Who to contact when things break</li></ul><h3>Why this matters:</h3><ul><li>Data pipelines break constantly</li><li>Knowing what’s expected helps debug faster</li><li>Clear contracts prevent assumptions</li></ul><h3>Evaluation reports:</h3><h3>Offline metrics:</h3><ul><li>What you measured</li><li>On what data</li><li>With what results</li><li>What the confidence intervals are</li></ul><h3>Online metrics:</h3><ul><li>What you measured in production</li><li>A/B test results</li><li>User feedback</li><li>Business impact</li></ul><h3>The delta:</h3><ul><li>Why offline and online differed</li><li>What you learned</li><li>What you’d do differently next time</li></ul><p><strong>Why this matters:</strong></p><ul><li>Offline metrics will lie to you</li><li>Documenting the gap helps you learn</li><li>Future experiments can learn from past ones</li></ul><h3>Decision logs:</h3><p><strong>Why you chose this:</strong></p><ul><li>Why this architecture?</li><li>Why this loss function?</li><li>Why this deployment strategy?</li><li>What alternatives did you consider?</li><li>What was the tradeoff?</li></ul><p><strong>What you expected:</strong></p><ul><li>What did you think would happen?</li><li>What did you optimize for?</li><li>What did you sacrifice?</li></ul><p><strong>What actually happened:</strong></p><ul><li>Were you right?</li><li>What surprised you?</li><li>What would you change?</li></ul><p><strong>Why this matters:</strong></p><ul><li>You’ll forget your reasoning</li><li>Other people need context</li><li>Decisions that seem obvious now will seem insane later (without context)</li><li>Learning happens when you compare expectations to reality</li></ul><p>All of these “forgotten” pieces have something in common:</p><p><strong>They’re not about making models work. They’re about keeping them working.</strong></p><ul><li>Observability: helps you debug when things break</li><li>Agents: handle autonomous failures gracefully</li><li>Security: prevent attacks before they happen</li><li>Documentation: preserve knowledge across time</li></ul><p><strong>The model is the easy part. Everything around it is the actual engineering.</p><h3>Tomorrow: Part 6</h3><p>Evals and organizational reality (what actually determines success.)</p><p>Why rigorous evaluation is the difference between a demo and a product. How to measure what matters. And why even perfect engineering fails in broken organizations.</p>
