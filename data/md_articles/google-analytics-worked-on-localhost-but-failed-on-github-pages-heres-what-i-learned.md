<p><em>A beginner-friendly debugging story about static sites, Next.js, Google Analytics, and why production can behave very differently from local development.</em></p><figure><img alt="" src="/images/blog/google-analytics-worked-on-localhost-but-failed-on-github-pages-heres-what-i-learned/hero.jpeg" /></figure><p>I think the problem was actually obvious.</p><p>But since I’m still quite new to web development, I had to go through the whole confusion → debugging → learning cycle. That’s also why I’m writing this.</p><p>So here’s what happened.</p><p>I was trying to add Google Analytics to my portfolio site. It’s built using Next.js and deployed on GitHub Pages.</p><p>On localhost, Google Analytics was detecting page visits. On the live .github.io site, it was silent.</p><p>At first, that made no sense. Same code, different results. It turns out that is completely possible, and understanding why is one of the more useful lessons you can pick up early.</p><p>This post walks through what happened, how I debugged it, and the broader engineering takeaway that I think applies well beyond Google Analytics.</p><h3>The Setup</h3><p>My site is built with Next.js, exported as a static site, and deployed to GitHub Pages.</p><p>I wanted something simple: track visitors, see daily and weekly numbers, get a sense of overall traffic over time. Google Analytics 4 was the obvious pick. Free, widely used, and more than enough for a personal site.</p><p>So I created a GA4 property, got the measurement ID, and added the standard snippet. The plan was straightforward:</p><ol><li>Load Google’s gtag.js</li><li>Initialize it with the measurement ID</li><li>Deploy the site</li><li>Watch traffic appear in GA Realtime</li></ol><p>That is what should have happened.</p><h3>The Confusing Part</h3><p>The first clue that something was off: localhost visits were showing up in Google Analytics, but GitHub Pages visits were not.</p><p>That made the problem genuinely puzzling.</p><p>If nothing worked at all, the suspects are obvious: wrong measurement ID, wrong GA property, missing script, bad environment variable. But because localhost was working, the GA account itself was clearly configured correctly. The problem lived somewhere between how the script was added, how the site was built, and how the static output behaved on GitHub Pages.</p><p>That narrowed things down quite a bit.</p><h3>The First Important Lesson</h3><p>When debugging production issues, looking at source code and saying ‘this looks fine’ is only half the job. What actually matters is what the browser receives and executes.</p><p>That shift in thinking is one of the more useful things I picked up from this whole experience.</p><p>Modern frameworks like Next.js do a lot behind the scenes. They optimize scripts, move things around, hydrate pages, and generate different output for development versus production. Your React or Next.js code can look perfectly correct while the final generated HTML behaves differently than expected. On static hosting, that difference matters a lot.</p><h3>What I Checked First</h3><p>The obvious things first:</p><ul><li>Correct GA measurement ID? Yes.</li><li>Correct environment variable name? Yes.</li><li>GitHub Actions workflow had access to the value? Yes.</li><li>Code pushed to main and the production site rebuilt? Yes.</li></ul><p>All fine. Still nothing in Realtime.</p><p>So I moved to browser-level checks.</p><h3>The Browser Checks That Helped</h3><p>On the live site, I opened the browser console and ran:</p><pre>typeof gtag</pre><p>I expected &#39;function&#39;. I got &#39;undefined&#39;.</p><p>That was a very useful clue. It meant the Google Analytics function had never been properly initialized in the browser.</p><p>Then I checked whether the Google script loader was even present:</p><pre>document.querySelector(&#39;script[src*=&quot;googletagmanager&quot;]&#39;)?.src</pre><p>That did exist.</p><p>So the picture became much clearer: the external Google script was loading, but the initialization step was failing silently. The loader was there; the initialized function was not.</p><h3>Why This Was Happening</h3><p>The original setup used framework-managed script injection inside the Next.js layout. That looked reasonable in code, and it worked locally. But in the final static export, the initialization script was behaving unreliably on the live GitHub Pages deployment.</p><p>There was no error message explaining this. No warning about static hosting constraints. Just silence.</p><p>That silence is what makes production debugging feel mysterious, especially early on.</p><h3>The Fix</h3><p>The reliable solution turned out to be simpler than expected: instead of relying on inline or framework-managed initialization, I switched to plain external scripts.</p><p>The final production setup:</p><ol><li>Load Google’s analytics script</li><li>Load a small custom external script, ga-init.js</li><li>Initialize gtag from there</li></ol><p>The browser now receives two normal script tags in the generated HTML:</p><pre>&lt;script async src=&quot;https://www.googletagmanager.com/gtag/js?id=G-XXXXXXXXXX&quot;&gt;&lt;/script&gt;<br>&lt;script defer src=&quot;/ga-init.js?id=G-XXXXXXXXXX&quot;&gt;&lt;/script&gt;</pre><p>And the initializer file:</p><pre>(function () {<br>  var currentScript = document.currentScript;<br><br>  if (!currentScript) {<br>    return;<br>  }<br><br>  var scriptUrl = new URL(currentScript.src, window.location.href);<br>  var measurementId = scriptUrl.searchParams.get(&#39;id&#39;);<br><br>  if (!measurementId) {<br>    return;<br>  }<br><br>  window.dataLayer = window.dataLayer || [];<br><br>  function gtag() {<br>    window.dataLayer.push(arguments);<br>  }<br><br>  window.gtag = gtag;<br>  gtag(&#39;js&#39;, new Date());<br>  gtag(&#39;config&#39;, measurementId);<br>})();</pre><p>After deploying that version, things started working correctly.</p><h3>Why the External Script Approach Helped</h3><p>For a static site on a simple host like GitHub Pages, making important browser behavior plain and explicit tends to work better. Fewer framework-specific layers, fewer assumptions about runtime behavior, more standard browser script loading.</p><p>Framework abstractions are genuinely useful. But for functionality that is simple and critical, the most reliable version is often the one closest to basic HTML and JavaScript.</p><p>That was the case here.</p><h3>The Bigger Lesson: Localhost Is a Comfortable Lie</h3><p>Just because something works on localhost does not mean it works in production.</p><p>That sounds obvious. But when you are actively building something, local success is easy to over-trust.</p><p>Local development and production can diverge for all kinds of reasons: build-time optimizations, static export behavior, script ordering, hydration differences, environment variables, caching, browser policies, and hosting platform constraints. Any one of these can produce a gap between what you see locally and what users actually get.</p><p>A good habit to build early: always verify the deployed version, not just the local version.</p><h3>The Debugging Process That Actually Helped</h3><p>Here is a summary of the approach that worked:</p><p><strong>Confirm what is already working.</strong> Localhost traffic appeared in GA, which told me the measurement ID and Analytics property were probably fine. That saved time by ruling out the wrong suspects early.</p><p><strong>Check the browser, not just the source code.</strong> Running typeof gtag in the console told me initialization had failed. One line, immediate clarity.</p><p><strong>Separate script presence from script execution.</strong> The Google loader script existed. The initialized function did not. A script tag being present in the DOM does not automatically mean the feature is working.</p><p><strong>Inspect the built output.</strong> Instead of asking ‘does my Next.js code look correct,’ the better question is ‘what exactly is in the exported HTML?’ That question gets much closer to the actual problem.</p><p><strong>Prefer the simpler runtime path when debugging static sites.</strong> When the abstraction is unclear, move closer to plain browser behavior. That is what solved this.</p><h3>What I Would Tell Someone Doing This for the First Time</h3><p>If you are adding Google Analytics to a static site:</p><ul><li>Verify the correct measurement ID is being used</li><li>Confirm the production build actually receives that value</li><li>Check the live deployed HTML, not just your local files</li><li>Use the browser console to confirm gtag really exists as a function</li><li>If framework-managed script injection behaves strangely, simplify the setup</li></ul>
